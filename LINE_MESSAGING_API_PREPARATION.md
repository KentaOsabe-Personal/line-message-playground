# LINE配信アプリを開発する前の準備

調査日: 2026-07-11  
対象: 個人学習として、LINE Messaging APIを使い任意のタイミングでメッセージを配信するWebアプリ  
参照方針: LINE DevelopersおよびLINEヤフー for Businessの公式情報

## 結論

最初のテスト配信までに必要な作業は、次の6点です。

1. 個人のLINEアカウントまたはメールアドレスでビジネスIDを登録し、LINE公式アカウントを作成する。
2. LINE Official Account ManagerからMessaging APIを有効化する。
3. 作成時に、チャネルを所属させる「プロバイダー」を慎重に選ぶ。
4. LINE Developersコンソールでチャネルアクセストークンを発行する。
5. 自分のLINEアカウントを公式アカウントの友だちに追加し、テスト送信先を用意する。
6. 料金プランと月間配信上限を確認してから、バックエンドから送信APIを呼ぶ。

全友だちへのブロードキャストだけを行う最小構成では、Webhookは必須ではありません。個別配信先のユーザーIDを友だち追加時に収集する、ユーザーの操作に応答する、といった機能も学ぶ段階で、公開HTTPS URLを持つWebhook受信環境を追加します。

### ビジネスIDとは

ビジネスIDは、法人番号、開業届、資格、審査済みの事業者アカウントではありません。LINEヤフーが提供するビジネス向け／開発者向け管理画面へログインするための**共通認証システム**です。

次のいずれかをログイン手段として登録できます。

- 普段利用している個人のLINEアカウント
- メールアドレスだけで作るビジネスアカウント
- 対応サービスではYahoo! JAPAN ID

名称に「ビジネス」とありますが、個人開発者も登録できます。公式案内でも、LINE公式アカウントの「未認証アカウント」は個人・法人を問わず誰でも取得でき、個人のLINEアカウントまたはメールアドレスから登録できると明記されています。Messaging APIの学習には未認証アカウントで十分です。

なお、2025年6月25日以降、新しいLINE公式アカウントの開設時にはビジネスマネージャーの「組織」への接続が必須です。個人学習でも作成画面にこの手順が表示されます。これはアプリの本番／開発環境を分ける話ではなく、LINEヤフー側のアカウント管理上必要な設定です。画面上の登録内容と規約を確認し、実態と異なる事業者情報は入力しないでください。

## 1. まず決めること

### 配信対象と送信方法

Messaging APIには主に次の送信方法があります。

| やりたいこと | 送信方法 | 事前に必要なもの |
| --- | --- | --- |
| 友だち全員へ配信 | ブロードキャスト | 友だち、チャネルアクセストークン |
| 1人へ配信 | プッシュ | 対象のユーザーID |
| 複数の指定ユーザーへ配信 | マルチキャスト | 対象者のユーザーID一覧 |
| 属性・オーディエンスで絞って配信 | ナローキャスト | 対象条件またはオーディエンス |
| ユーザー操作へ返信 | 応答メッセージ | Webhookで受け取る応答トークン |

本プロジェクトの最初のスコープは、まず「自分へのプッシュ送信」または「テスト用公式アカウントの全友だちへのブロードキャスト」にすると確認しやすいです。

### この学習で使う構成

このプロジェクトでは、本番用／開発用を分けず、学習用のLINE公式アカウントとMessaging APIチャネルを1つだけ作ります。友だちは自分だけにして、自分のLINEアプリへテスト配信します。テストでも実際のLINE公式アカウントと無料メッセージ通数を使用します。

## 2. アカウントとチャネルの準備

### 2-1. ビジネスIDとLINE公式アカウントを作成する

Messaging APIを利用するにはLINE公式アカウントが必要です。公式の開始手順は次の順序です。

1. LINEアカウントまたはメールアドレスでビジネスIDに登録する。
2. LINE公式アカウントの作成フォームへ必要事項を入力する。
3. LINE Official Account Managerで作成されたアカウントを確認する。

現在の新規開設フローでは、途中でビジネスマネージャーの組織を新規作成するか、既存組織へ接続します。個人学習では既存組織がなければ画面の案内に沿って新規作成します。

既存のLINE公式アカウントがある場合は、新規作成せず次の「Messaging APIを有効化する」作業から開始できます。

### 2-2. Messaging APIを有効化する

LINE Official Account Managerから対象アカウントのMessaging APIを有効にします。これによりMessaging APIチャネルが作成されます。

> 現在、LINE DevelopersコンソールからMessaging APIチャネルを直接新規作成することはできません。LINE公式アカウントを作成し、Official Account Managerから有効化する必要があります。

### 2-3. プロバイダーを選択する

有効化の途中で、チャネルを所属させるプロバイダーを選びます。ここは事前に運用主体を決めてから実施します。

- 既存プロバイダーを選ぶには、そのプロバイダーのAdmin権限が必要。
- 一度選ぶと、後から別プロバイダーへ変更したりMessaging APIとの連携を解除したりできない。
- ユーザーIDはプロバイダー単位で発行される。同じユーザーでも異なるプロバイダーでは異なるIDになる。
- 将来LINEログイン等と連携するなら、原則として同じサービスを同じプロバイダー配下に置く。
- 制作会社や外部ツール事業者ではなく、サービスを継続運用する組織が管理できる構成を検討する。

### 2-4. 個人開発での権限

個人学習では自分だけを管理者にすれば十分です。チャネルシークレットの取得・再発行にはAdmin権限が必要なので、自分のアカウントがAdminになっていることだけ確認します。

## 3. 認証情報の準備

### チャネルアクセストークン

アプリがMessaging APIを呼ぶための認証情報です。公式ドキュメントでは、最大30日の有効期間を指定でき、JWTを用いる「チャネルアクセストークンv2.1」が推奨されています。

主な選択肢は次のとおりです。

| 種類 | 有効期間 | 主な用途・注意 |
| --- | --- | --- |
| v2.1（推奨） | 最大30日で任意指定 | JWTを用いて発行。更新処理も含めた発展学習向け |
| ステートレス | 15分 | 短時間利用向け。発行数は無制限 |
| 短期 | 30日 | チャネルごとに最大30件 |
| 長期 | 無期限 | コンソールから発行可能だが、1チャネル1件。漏えい時の影響が長い |

学習の最初の疎通確認は、コンソールから発行しやすい長期トークンで進められます。トークンv2.1と自動更新は、基本の配信に成功した後の発展課題にします。

### チャネルシークレット

Webhookを使う場合、LINEから届いたリクエストの署名検証に必要です。送信APIを呼ぶだけの処理では使用しません。

### このプロジェクトでの管理方針

- トークン、チャネルシークレット、秘密鍵はDjangoバックエンドだけが参照する。
- Reactフロントエンド、ブラウザ、APIレスポンスへ渡さない。
- Gitへコミットせず、ローカルでは`.env`へ保存する。将来どこかへ公開する場合も、ソースコードではなく公開先の環境変数等へ保存する。
- ログへトークンや署名検証前の機微な本文を不用意に出力しない。
- 漏えいが疑われたら直ちにトークンの取り消し／再発行を行う。
- 学習用でも、漏えい時に再発行できることを確認しておく。

実装時に追加する環境変数の例（値は記載しない）:

```dotenv
LINE_CHANNEL_ACCESS_TOKEN=
LINE_CHANNEL_SECRET=
```

## 4. 配信先の準備

### 友だち追加

LINE Developersコンソールの［Messaging API設定］に表示されるQRコードから、開発者自身のLINEアカウントを公式アカウントの友だちに追加します。ブロックしているユーザーには配信されません。

### ユーザーID

個別のプッシュ／マルチキャスト配信には、LINE IDや表示名ではなく、LINEプラットフォームが発行するユーザーIDが必要です。

取得方法は主に次の4つです。

1. 開発者本人のIDをLINE Developersコンソールの［チャネル基本設定］→［あなたのユーザーID］で確認する。
2. 友だち追加時などのWebhookイベントから`source.userId`を保存する。
3. 友だち全員のユーザーID取得APIを利用する。
4. グループ／複数人トークのメンバーID取得APIを利用する。

注意点:

- 友だち全員のユーザーID取得APIは、認証済アカウントまたはプレミアムアカウントでのみ利用可能。
- WebhookでユーザーIDを継続収集する設計が、未認証の開発用アカウントでも扱いやすい。
- ユーザーIDは個人に結び付くデータとして、利用目的、アクセス権、保持期間、削除方法を決める。
- LINEユーザーデータポリシー、LINE開発者契約、LINE公式アカウント関連規約を開発前に確認する。

## 5. Webhookを使う場合の追加準備

全友だちへの一方向配信だけなら後回しにできます。個別ユーザー管理や応答機能を作る場合は準備します。

### 公開HTTPSエンドポイント

- LINE Developersコンソールに設定できるWebhook URLは1チャネルにつき1つ。
- URLはインターネットから到達可能なHTTPSである必要がある。
- 公的に信頼された認証局の証明書が必要で、自己署名証明書は利用できない。
- `http://localhost:8000`へLINEから直接アクセスすることはできない。開発時はHTTPSトンネルまたは検証環境へのデプロイが必要。
- 疎通確認では、`events`が空のHTTP POSTにも`200`を返す。
- コンソールの［検証］で成功を確認後、［Webhookの利用］を有効にする。

### 署名検証

受信した生のリクエストボディとチャネルシークレットを使い、`x-line-signature`をHMAC-SHA256で必ず検証してからJSONを処理します。検証前にボディをデシリアライズ、整形、文字列置換しないようにします。送信元IPアドレスは公開されていないため、IP許可リストの代わりに署名検証を使います。

### 再送と非同期処理

- Webhook処理は速やかに2xxを返し、重い処理は非同期化することが公式に推奨されている。
- Webhook再送はデフォルトで無効。必要ならコンソールで有効化する。
- 再送やネットワーク事情で同じイベントが複数回来る可能性があるため、`webhookEventId`で重複を排除する。
- 再送時は順番が入れ替わる可能性があるため、必要に応じて`timestamp`も確認する。

### 自動応答設定

LINE Official Account Managerの「あいさつメッセージ」「応答メッセージ」は初期状態で有効です。Messaging API側でも同じ応答を実装すると二重送信になり得ます。初期開発では両方をオフにすることが公式に推奨されています。意図して併用する場合だけ有効にします。

## 6. 料金・配信上限の確認

配信開始前に、LINE Official Account Managerで料金プランと当月の上限を確認します。2026-07-11時点で公式ドキュメントに掲載されている日本向け例は次のとおりです（税別。料金は変更され得るため契約時に再確認してください）。

| プラン | 月額固定費 | 無料メッセージ通数／月 | 追加メッセージ |
| --- | ---: | ---: | --- |
| コミュニケーション | 0円 | 200通 | 不可 |
| ライト | 5,000円 | 5,000通 | 不可 |
| スタンダード | 15,000円 | 30,000通 | ～3円／通 |

通数は「1リクエストで指定した吹き出し数」ではなく、基本的に送信対象人数で数えます。1回のリクエストに複数のメッセージオブジェクトを含めても、その件数自体は通数に影響しません。

- 課金通数に含まれる: プッシュ、マルチキャスト、ブロードキャスト、ナローキャスト。
- 課金通数に含まれない: 応答メッセージ。
- ブロック済み／存在しないIDなど、実際に届かなかった対象は通数に含まれない。
- 上限超過時はAPIがエラーとなり、配信されない。
- APIから当月の上限目安と利用状況を取得できるため、配信直前のチェックと監視に利用する。

アプリ側にも、対象人数の事前表示、確認画面、二重送信防止、月間上限チェック、監査ログを用意するのが望ましいです。

## 7. 学習用テストの進め方

### 最小の疎通確認

1. 学習用LINE公式アカウントを作成し、Messaging APIを有効にする。
2. 開発者自身を友だち追加する。
3. チャネルアクセストークンを発行し、バックエンドの環境変数へ設定する。
4. コンソールにある開発者自身のユーザーIDを宛先に、テキスト1件をプッシュ送信する。
5. APIのHTTPステータス、`X-Line-Request-Id`、受信結果を確認する。
6. 同じ操作を連打して二重送信しないことを確認する。

### 学習開始チェック

- [ ] 運用主体に合ったプロバイダーを選択した。
- [ ] 学習用の公式アカウントを1つ作り、友だちは自分だけにした。
- [ ] トークンとチャネルシークレットをGitやフロントエンドへ置いていない。
- [ ] 配信対象と送信方法（push / multicast / broadcast / narrowcast）を決めた。
- [ ] 友だち獲得方法と、個別配信する場合のユーザーID取得方法を決めた。
- [ ] 料金プラン、月間上限、想定対象人数、予算を確認した。
- [ ] ボタン連打などによる重複送信を防ぐ方法を決めた。
- [ ] APIエラー、429、上限超過、タイムアウト時の扱いを決めた。
- [ ] 大量リクエストや同一ユーザーへの大量送信を行わない。
- [ ] LINEプラットフォーム経由で負荷テストを行わない。
- [ ] 学習に不要なユーザーデータを保存しない。
- [ ] Webhook利用時は公開HTTPS、署名検証、重複排除、再送設定を確認した。
- [ ] あいさつ／自動応答の二重送信がないことを確認した。
- [ ] 自分だけを対象にテスト配信し、表示とリンクを実機確認した。

## 8. このプロジェクトで次に実装する内容

現在の構成はReactフロントエンド、Django REST Frameworkバックエンド、MySQLです。次の実装順が適しています。

1. ComposeからDjangoへ`LINE_CHANNEL_ACCESS_TOKEN`を渡せるようにする。
2. DjangoからLINEのプッシュまたはブロードキャストAPIを呼ぶサービス層を作る。
3. まず固定のテストメッセージを自分だけに送る管理用APIを作る。
4. 送信要求に一意キーを付け、二重送信を防ぐ。
5. 配信内容、対象、日時、LINEのリクエストID、結果をDBへ記録する。
6. フロントエンドに確認画面を設け、ブラウザからLINEのAPIを直接呼ばない。
7. 必要になった段階でWebhook受信、署名検証、ユーザーID管理を追加する。
8. 配信前の月間利用状況確認、失敗時の再試行方針、監視・通知を追加する。
