# プロダクト概要

LINE Message Playground は、個人開発者が LINE Messaging API による配信を、実アカウントを使った小さく安全な構成から段階的に学ぶためのローカル検証環境です。

最初から本番向けの配信管理機能を網羅するのではなく、自分宛ての送信を再現可能かつ追跡可能にし、その後に必要な能力だけを追加します。

現在は基盤拡張の段階にあり、固定設定による配信に加えて、複数 Messaging API チャネルの資格情報管理、LIFF／LINE Login による本人連携、検証済み Webhook 受付、follow／unfollow による既存配信先の友だち状態同期、message／postback の許可リスト型処理までを備えています。登録済み配信先への送信は、この信頼境界の上へ段階的に追加します。

## 対象ユーザー

- LINE Messaging API の送信・認証・Webhook・運用制約を実践的に学ぶ個人開発者
- 学習用 LINE 公式アカウントを自分で管理し、まず自分自身を送信対象にできる利用者

複数事業者向けの運用、大規模キャンペーン、高度な顧客管理は初期スコープに含めません。

## 中核能力

### 開発環境の疎通確認

フロントエンド、バックエンド、データベースを一体で起動し、ブラウザからバックエンド API までの接続状態を確認します。

### 安全なテスト配信

既存の配信経路は Backend 設定で固定した自分宛てのテキスト push を扱います。登録済み配信先を選ぶ経路へ移行するまでは、ブラウザから宛先を変更させません。どちらの経路でも、入力内容を整形したプレビューと配信対象を確認してから送信することを重視します。broadcast は必要性と対象範囲を確認した後の拡張候補です。

### 自分専用の認証とアカウント連携

LIFF／LINE Login の結果を Backend で検証し、許可した本人だけを owner session に結び付けます。LINE identity とチャネル別配信先の登録・解除を本人操作として扱い、セッション失効、連携解除の中断、再認証から安全に回復できることを重視します。

### 秘密情報の隔離

チャネルアクセストークンやチャネルシークレットは認証付き暗号で暗号化してデータベースへ保存し、バックエンドだけが必要時に復号します。暗号化マスターキーと LINE Login の秘密値は環境変数に隔離し、ブラウザ、React、API レスポンス、Git、ログへ露出させません。

### 冪等性と監査可能性

配信は取り消せない外部作用として扱います。一意キーによる二重送信防止と、対象・内容・日時・結果・LINE のリクエスト ID を追跡できる設計を基本とします。タイムアウト等で結果が不明な場合は成功や失敗を推測せず、同じ操作の状態を確認してから安全な再試行可否を判断します。

### 段階的な機能拡張

一方向配信だけの段階では Webhook を導入せず、個別ユーザーの状態同期や応答が必要になってから追加します。Webhook 受付境界は、公開 HTTPS、チャネル選択、署名と destination の検証、イベント重複排除、安全な監査までを担当します。その下流では、検証済み follow／unfollow だけを時系列どおり既存配信先へ反映し、未連携ユーザーを登録しません。

検証済み message／postback は、完全一致の静的許可リストと既存の本人連携を通過した場合だけ処理します。現在の command は疎通確認用の固定 reply に限定し、postback は登録済み action へ安全な値だけを渡す拡張境界とします。未知・不正・未連携の入力から利用者や配信先を作らず、外部作用なしの結果へ収束させます。

## 主要ユースケース

- ローカル環境を起動し、Frontend から Backend API への接続を確認する
- Django 経由で自分の LINE アプリへテストメッセージを送り、実機表示と API 結果を確認する
- LIFF／LINE Login で本人確認し、同一 provider 配下のチャネル別配信先を登録・解除する
- 複数チャネルの資格情報を暗号化して管理し、チャネルごとの Webhook を安全に受け付ける
- 検証済み follow／unfollow から既存配信先の友だち状態を同期し、配信可否を確認する
- 検証済み message／postback を静的許可リストで処理し、限定 reply または登録済み action へ一回だけ委譲する
- 送信前の確認と冪等化により、連打や再試行による二重配信を防ぐ
- 配信結果を確認し、失敗や結果不明を安全に扱う
- 将来は配信履歴と利用量を確認し、月間上限を安全に扱う
- 必要に応じて送信方式、Webhook、ユーザー ID 管理へ学習範囲を広げる

## 価値提案

- 実際の LINE Messaging API の制約を、UI とバックエンドの両方で学べる
- 最小の成功体験から始め、秘密管理、冪等性、監査、個人データ保護へ段階的に進める
- 機能数ではなく、安全に再現でき、結果を追跡できることを成功の基準にする

## プロダクト原則

- 学習に不要な LINE ユーザーデータは保存しない
- ユーザー ID を保存する場合は、目的・アクセス権・保持期間・削除方法を決める
- 対象人数、料金プラン、月間上限を送信前に確認できるようにする
- LINE プラットフォーム経由の負荷試験や、不特定多数への試験配信を行わない
- 外部 API の失敗、429、タイムアウトを正常系とは別の明示的な状態として扱う
- 結果不明を成功扱いせず、状態確認前に別の新規送信を始めない
