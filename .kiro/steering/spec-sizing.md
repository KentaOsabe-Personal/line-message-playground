# Specサイズと分割の方針

## 目的

タスク生成まで進んでからSpecが大きすぎると判明することを防ぎます。単一Specは、1つの明確な成果と責任境界を持ち、レビュー可能な実装単位に収まる必要があります。

## 判定結果

各フェーズで、次のどちらかを明示します。

- `PASS (single-spec)`: 単一Specとして継続できる
- `SPLIT_REQUIRED`: Discoveryへ戻り、roadmap上で複数Specへ分割する

判定には、少なくとも次の根拠を含めます。

- 独立した責任境界の候補
- 独立して提供・変更・レビューできる成果の数
- 外部サービスとの連携や状態遷移を伴う独立ワークフローの数
- 1〜3時間単位で見積もった実行可能タスク数（テスト、移行、統合を含む）
- 判定理由

## `SPLIT_REQUIRED` の基準

次のいずれかに該当する場合は、原則として分割します。

1. 実行可能タスクが20件以上になる見込みがある
2. 独立して提供・変更・レビューできる責任境界が複数ある
3. 独立した状態機械、外部整合性処理、または失敗時の補償フローを複数含む
4. 共通・上流基盤の拡張と、その基盤を利用する下流機能の所有を同じSpecへ取り込んでいる
5. 多数のタスクが広い `_Boundary:_` を必要とする、または境界をまたぐ統合作業が繰り返される

20件は目標値ではなく停止基準です。19件以下であっても、独立した責任境界が複数あれば分割します。

一方、UI・API・永続化をすべて変更するという理由だけでは分割しません。同じ成果と責任境界を実現する垂直スライスであれば、単一Specにできます。

## フェーズごとのゲート

### Discovery

- Path C（単一Spec）を確定する前にサイズ判定を行う
- 判定結果とタスク数の見積りを `brief.md` の `Spec Size Assessment` に記録する
- `SPLIT_REQUIRED` の場合はPath D/Eを選び、`roadmap.md` と各Specの `brief.md` を作成する

### Spec Init / Quick

- Discoveryを経由していなくても、ファイル作成前にサイズ判定を行う
- サイズ判定がない、または `SPLIT_REQUIRED` の場合は作成を停止し、`$kiro-discovery` を案内する
- `--auto` はこのゲートを省略しない

### Requirements

- Requirementsのドラフトを保存する前に、ユーザージャーニー、責務、外部ワークフローからタスク数を再見積りする
- `SPLIT_REQUIRED` の場合は `requirements.md` を確定せず、Discoveryへ戻す

### Design

- コンポーネント、状態遷移、統合点、移行、テスト戦略が明らかになった時点で再判定する
- 判定根拠を `research.md` に記録する
- `SPLIT_REQUIRED` の場合は `design.md` を確定せず、roadmap分割へ戻す

### Tasks

- ドラフト中の実行可能タスクを数え、最終バックストップとして判定する
- 20件以上、または境界横断が反復する場合は `tasks.md` を書かずに停止する
- タスクを不自然に結合して件数だけ減らしてはならない

## 例外

ユーザーがトレードオフを理解したうえで単一Specの継続を明示した場合のみ例外を認めます。その場合は、判定結果、継続理由、レビュー・実装上のリスクを `brief.md` または `research.md` に記録します。自動モードや暗黙の判断で例外扱いにしてはなりません。

この方針は新規作成・再生成時に適用し、既存Specを自動的に分割または書き換えるものではありません。
