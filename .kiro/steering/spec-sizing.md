# Specサイズと分割の方針

## 目的

タスク生成まで進んでからSpecが大きすぎると判明し、レビュー修正を何度も繰り返すことを防ぎます。単一Specは、複数の内部ワークストリームを含んでもよい一方、要件・設計・タスクを一つの成果として安定してレビューできる実装単位に収まる必要があります。

## 判定結果

各フェーズで、次のどちらかを明示します。

- `PASS (single-spec)`: 単一Specとして継続できる
- `SPLIT_REQUIRED`: Discoveryへ戻り、roadmap上で複数Specへ分割する

判定には、少なくとも次の根拠を含めます。

- 独立した責任境界の候補
- 独立して提供・変更・レビューできる成果の数
- 外部サービスとの連携や状態遷移を伴う独立ワークフローの数
- 1〜3時間単位で見積もった実行可能タスク数（テスト、移行、統合を含む）
- 判定理由

## `SPLIT_REQUIRED` の基準

次のいずれかに該当する場合は、原則として分割します。

1. 実行可能タスクが40件以上になる見込みがある
2. 独立して提供・変更・レビューできる成果が複数あり、さらに独立した状態機械／補償フロー、上流基盤と下流利用機能の同居、反復する境界横断統合、または別々のrollout／rollbackのうち2つ以上が重なる
3. Requirements、Design、Tasksのbounded reviewで、同じ構造的問題（責任重複、隠れた前提、依存矛盾、広すぎるtask）が修正上限まで解消せず、単一レビュー範囲として収束しない
4. 1〜3時間の実行単位を維持すると広い `_Boundary:_` や境界横断integration taskが多数必要になり、逆に件数を減らすと前提・検証・失敗処理を隠すことになる

件数はレビュー負荷の指標であり、それだけを分割目的にしません。目安は次のとおりです。

- 29件以下: 通常の単一Spec候補。複数の内部境界があっても、それだけでは分割しない
- 30〜39件: review attention帯。内部ワークストリーム、依存順、統合点、検証戦略を明示し、bounded reviewが収束すれば単一Specを許容する
- 40件以上: 原則 `SPLIT_REQUIRED`。件数を不自然に減らさず、ユーザー承認例外がなければ分割する

独立した責任境界が複数あること、またはUI・API・永続化をすべて変更することだけでは分割しません。同じ利用者成果へ収束し、境界ごとの所有と依存順が明確で、レビューがbounded pass内に収束するなら単一Specにできます。

一方、UI・API・永続化をすべて変更するという理由だけでは分割しません。同じ成果と責任境界を実現する垂直スライスであれば、単一Specにできます。

## フェーズごとのゲート

### Discovery

- Path C（単一Spec）を確定する前にサイズ判定を行う
- 判定結果とタスク数の見積りを `brief.md` の `Spec Size Assessment` に記録する
- `SPLIT_REQUIRED` の場合はPath D/Eを選び、`roadmap.md` と各Specの `brief.md` を作成する

### Spec Init / Quick

- Discoveryを経由していなくても、ファイル作成前にサイズ判定を行う
- サイズ判定がない、または `SPLIT_REQUIRED` の場合は作成を停止し、`$kiro-discovery` を案内する
- `--auto` はこのゲートを省略しない

### Requirements

- Requirementsのドラフトを保存する前に、ユーザージャーニー、責務、外部ワークフローからタスク数を再見積りする
- 30〜39件の見積りでは、内部ワークストリームと依存順を記録し、件数だけで分割しない
- `SPLIT_REQUIRED` の場合は `requirements.md` を確定せず、Discoveryへ戻す

### Design

- コンポーネント、状態遷移、統合点、移行、テスト戦略が明らかになった時点で再判定する
- 判定根拠を `research.md` に記録する
- 複数境界を持つ場合は、file owner、contract、integration task、review順序を明示し、bounded reviewが収束するかを判定する
- `SPLIT_REQUIRED` の場合は `design.md` を確定せず、roadmap分割へ戻す

### Tasks

- ドラフト中の実行可能タスクを数え、最終バックストップとして判定する
- 40件以上、またはbounded reviewで構造的問題が修正上限まで収束しない場合は `tasks.md` を書かずに停止する
- 30〜39件は独立task-graph sanity reviewを必須とし、`PASS`へ収束すれば書き込みを許容する
- タスクを不自然に結合して件数だけ減らしてはならない

## 例外

ユーザーがトレードオフを理解したうえで単一Specの継続を明示した場合のみ例外を認めます。その場合は、判定結果、継続理由、レビュー・実装上のリスクを `brief.md` または `research.md` に記録します。自動モードや暗黙の判断で例外扱いにしてはなりません。

この方針は新規作成・再生成時に適用し、既存Specを自動的に分割または書き換えるものではありません。
