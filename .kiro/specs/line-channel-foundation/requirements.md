# 要求仕様書

## はじめに

LINE Message Playground に、複数の LINE 公式アカウントに対応する Messaging API チャネルと、そのアクセストークンおよびチャネルシークレットを安全に管理する共通基盤を追加する。個人開発者は管理画面が未完成でもチャネルを初期登録・更新・無効化でき、後続の配信処理と Webhook 処理は必要な資格情報だけを安全に取得できる。保存時の暗号化、秘密情報の非露出、設定不備の早期検出、中断可能な鍵ローテーションを必須とする。

## 境界コンテキスト

- **対象範囲**: 複数 Messaging API チャネルの識別情報と有効状態、アクセストークンとチャネルシークレットの暗号化保存、用途別の資格情報取得、専用暗号化キー設定の起動時検証、対話式管理コマンドによる登録・更新・有効化・無効化、段階的な鍵ローテーション、秘密情報のマスキング
- **対象外**: React 管理画面、LIFF／LINE ログイン、配信先管理、Webhook の受付・署名検証・イベント処理、可変宛先への配信、既存配信 API のチャネル選択対応、外部 KMS、複数管理者の認証・権限管理、保存済み資格情報の表示・コピー・エクスポート
- **隣接する前提**: 現行の固定宛先配信は、本仕様では環境変数から資格情報を取得する既存の資格情報取得契約を維持する。後続の配信仕様と Webhook 仕様は本基盤のチャネル識別情報および用途別資格情報取得を利用する。暗号化キーの安全な注入、DB 外でのバックアップ、および旧鍵を撤去する最終操作は運用者が担う。

## 要求事項

### Requirement 1: 複数チャネルの登録と状態管理

**目的:** 個人開発者として、複数の Messaging API チャネルを明確に区別して登録し、有効状態を管理したい。これにより、学習対象のチャネルを安全に追加・切り替えできる。

#### 受入基準

1. The LINEチャネル基盤 shall 複数の Messaging API チャネルを、それぞれ一意に識別できる情報、運用者向け名称、および有効状態とともに保持する
2. When 新しいチャネルを登録する, the LINEチャネル基盤 shall 公式アカウントと Messaging API チャネルを対応づける識別情報、作成日時、および更新日時を記録する
3. When 新しいチャネルを登録する, the LINEチャネル基盤 shall 内部の連番や資格情報を含まない一意で不透明な公開識別子を発行する
4. If 登録済みチャネルと同じ一意な識別情報が指定される, the LINEチャネル基盤 shall 既存チャネルを上書きせず、重複を示す安全なエラーを返す
5. When チャネルの非秘密メタデータ、運用者向け名称、または有効状態を更新する, the LINEチャネル基盤 shall 対象チャネルだけを更新し、更新日時を記録する
6. When チャネルを無効化する, the LINEチャネル基盤 shall そのチャネルを有効チャネルの取得対象から除外し、識別情報と保存済み資格情報を保持する
7. When 完全な資格情報を保持し、設定済み鍵で復号可能な無効チャネルを再び有効化する, the LINEチャネル基盤 shall 同じチャネル識別情報と保存済み資格情報を維持したまま有効チャネルの取得対象へ戻す

### Requirement 2: 資格情報の暗号化保存と完全性保護

**目的:** 個人開発者として、チャネル資格情報を平文のまま永続化せず、安全に保存したい。これにより、DB の内容が参照されても資格情報が直接露出しない。

#### 受入基準

1. When チャネルアクセストークンとチャネルシークレットを受け付ける, the LINEチャネル基盤 shall 両方を認証付き暗号で暗号化してから永続化する
2. The LINEチャネル基盤 shall チャネルアクセストークンまたはチャネルシークレットの平文を永続データとして保存しない
3. When 資格情報を正常に保存する, the LINEチャネル基盤 shall 資格情報の設定済み状態と更新日時を秘密値なしで追跡できるようにする
4. If 資格情報の暗号化に失敗する, the LINEチャネル基盤 shall 新規または更新後の資格情報を一切保存せず、安全なエラーとして処理する
5. If 保存済み暗号文が改変、破損、または欠損している, the LINEチャネル基盤 shall 完全性検証に失敗し、平文または部分的な資格情報を返さない
6. If チャネルアクセストークンまたはチャネルシークレットのいずれかが登録時に不足している, the LINEチャネル基盤 shall 不完全な資格情報を保存せず、登録を完了しない

### Requirement 3: 用途別の資格情報取得

**目的:** 後続機能の開発者として、対象チャネルと用途を指定し、必要な資格情報だけを取得したい。これにより、配信処理と Webhook 処理が余分な秘密情報へ触れずに済む。

#### 受入基準

1. When 有効チャネルの送信用資格情報を要求する, the LINEチャネル基盤 shall 対象チャネルのアクセストークンだけを復号して要求元へ渡す
2. When 有効チャネルの Webhook 検証用資格情報を要求する, the LINEチャネル基盤 shall 対象チャネルのチャネルシークレットだけを復号して要求元へ渡す
3. If 指定されたチャネルが存在しない, the LINEチャネル基盤 shall 資格情報を返さず、チャネルを利用できないことを示す安全なエラーを返す
4. If 指定されたチャネルが無効である, the LINEチャネル基盤 shall 資格情報を返さず、チャネルが無効であることを示す安全なエラーを返す
5. If 指定されたチャネルに完全な資格情報が設定されていない, the LINEチャネル基盤 shall 資格情報を返さず、設定不備を示す安全なエラーを返す
6. If 資格情報を復号または完全性検証できない, the LINEチャネル基盤 shall 資格情報を返さず、復旧が必要な設定エラーとして扱う
7. The LINEチャネル基盤 shall 復号した資格情報を再び永続化、通常ログへ記録、または後続要求のために保持しない

### Requirement 4: 専用暗号化キー設定の安全な検証

**目的:** 個人開発者として、資格情報を利用する前に暗号化キーの設定不備を把握したい。これにより、不正な設定のまま起動してデータを保存または利用することを防げる。

#### 受入基準

1. When Backend を起動する, the LINEチャネル基盤 shall 資格情報用の専用暗号化キー設定を資格情報操作より先に検証する
2. If 専用暗号化キー設定が未指定、空、または受け付けられない形式である, the LINEチャネル基盤 shall 起動を失敗させ、秘密値を含まない設定エラーを示す
3. The LINEチャネル基盤 shall 暗黙の既定鍵を生成または使用しない
4. The LINEチャネル基盤 shall Django の SECRET_KEY を資格情報用暗号化キーの代替またはフォールバックとして使用しない
5. The LINEチャネル基盤 shall 資格情報用暗号化キーをチャネル資格情報と同じ永続データへ保存しない
6. Where 複数の暗号化キーがローテーション期間中に設定されている, the LINEチャネル基盤 shall 現用鍵で新規および更新資格情報を保護し、移行前の設定済み鍵で保護された資格情報も読み取れるようにする
7. If どの設定済み鍵でも資格情報を復号できない, the LINEチャネル基盤 shall 復号失敗を黙殺せず、当該資格情報を利用不能として扱う

### Requirement 5: 対話式コマンドによる初期登録と更新

**目的:** 個人開発者として、管理画面がなくても対話式操作からチャネルと資格情報を安全に初期登録・更新したい。これにより、後続機能の開発を始められる。

#### 受入基準

1. When 新しいチャネルの登録操作を開始する, the LINEチャネル基盤 shall 対話式にチャネル識別情報、運用者向け名称、アクセストークン、チャネルシークレット、および初期有効状態を受け付ける
2. While アクセストークンまたはチャネルシークレットの入力を受け付ける, the LINEチャネル基盤 shall 入力値を端末へ表示しない
3. When 登録済みチャネルの更新操作を開始する, the LINEチャネル基盤 shall 対象チャネルを明示させ、メタデータ、資格情報、および有効状態のうち指定された内容だけを更新できるようにする
4. If 対話式操作が取り消される、入力検証に失敗する、または暗号化に失敗する, the LINEチャネル基盤 shall 対象チャネルを部分的な登録または更新状態にしない
5. If 更新対象のチャネルが存在しない, the LINEチャネル基盤 shall 新しいチャネルを暗黙に作成せず、安全なエラーを表示する
6. When 登録または更新が完了する, the LINEチャネル基盤 shall 対象の非秘密識別情報、有効状態、および完了結果だけを表示する
7. The LINEチャネル基盤 shall 対話式操作の入力、完了出力、またはエラー出力へ平文資格情報も暗号文も表示しない

### Requirement 6: 中断可能な暗号化キーローテーション

**目的:** 個人開発者として、中断可能な手順で暗号化キーを段階的に交換したい。これにより、保存済み資格情報を失わずに旧鍵を安全に撤去できる。

#### 受入基準

1. When 新しい現用鍵と移行前の鍵を設定してローテーションを開始する, the LINEチャネル基盤 shall 保存済みの全チャネル資格情報を新しい現用鍵で再暗号化する
2. When 各資格情報を再暗号化する, the LINEチャネル基盤 shall 元の資格情報を復号でき、新しい暗号文を再び正常に復号できることを確認してから当該更新を完了する
3. If 個別の資格情報を復号、再暗号化、または検証できない, the LINEチャネル基盤 shall その資格情報を未検証の新しい暗号文で上書きせず、対象の非秘密識別情報と安全な失敗理由を報告する
4. If ローテーションが途中で中断する, the LINEチャネル基盤 shall 新旧両方の鍵が設定されている間、処理済み資格情報と未処理資格情報を引き続き取得可能にする
5. When 中断したローテーションを再実行する, the LINEチャネル基盤 shall 検証済みの資格情報を破損させず、残りの資格情報を安全に処理できるようにする
6. When 全資格情報が新しい現用鍵で復号可能であることを検証する, the LINEチャネル基盤 shall ローテーション完了と旧鍵を撤去可能であることを秘密値なしで報告する
7. If 一件でも新しい現用鍵で復号できない資格情報が残る, the LINEチャネル基盤 shall ローテーション完了を報告しない
8. The LINEチャネル基盤 shall ローテーションの進捗、結果、またはエラー出力へ暗号化キー、平文資格情報、もしくは暗号文を含めない

### Requirement 7: 秘密情報の非露出と安全な運用出力

**目的:** 個人開発者として、日常的な操作や障害調査で資格情報を誤って露出させたくない。これにより、DB 以外の画面、応答、ログ、デバッグ出力からの漏えいを防げる。

#### 受入基準

1. The LINEチャネル基盤 shall 平文資格情報、暗号文、および暗号化キーをモデルの文字列表現、管理用表示、API レスポンス、通常ログ、例外メッセージ、またはデバッグ出力へ含めない
2. When 資格情報に関する操作結果を表示または記録する, the LINEチャネル基盤 shall 非秘密のチャネル識別情報、設定済み状態、日時、および安全な結果分類だけを使用する
3. If 下位処理から秘密情報を含む可能性があるエラーを受け取る, the LINEチャネル基盤 shall その内容をそのまま伝播せず、安全なエラー分類へ置き換える
4. The LINEチャネル基盤 shall 保存済みの平文資格情報または暗号文を読み出して表示、コピー、またはエクスポートする操作を提供しない
5. When 資格情報を更新する, the LINEチャネル基盤 shall 既存値を表示せず、新しく入力された値だけを置換対象として扱う
