# Implementation Plan

- [x] 1. Backendの配信基盤を準備する
- [x] 1.1 配信機能の実行環境とアプリケーション境界を有効化する
  - 固定バージョンのLINE SDKをBackend依存関係へ追加し、v3 APIだけを利用できる状態にする
  - 配信機能を独立したDjango appとして登録する
  - access tokenと固定宛先の既存Backend-only環境変数配線を変更せず利用可能であることを確認する
  - チャネルシークレットを配信処理の設定へ取り込まない
  - Djangoの起動確認で配信appが認識され、既存のhealth機能が引き続き利用できる状態にする
  - _Requirements: 3.1, 3.3, 6.4_

- [x] 1.2 送信試行と状態遷移を永続化する
  - 操作ID、入力内容、整形済みテキスト、内容fingerprint、固定宛先方式、受付・期限・完了日時、結果、LINE request IDを追跡できる送信試行を定義する
  - 操作IDと処理中内容の一意制約、および状態別の必須値を保証する制約を設ける
  - 処理中から成功・失敗・結果不明への一方向遷移、terminal状態の再更新拒否、処理中fingerprintの解放を保証する
  - 宛先値、認証情報、確認トークン、LINEのraw errorを保存または文字列表現へ含めない
  - 初期migrationを適用すると制約付きの送信試行テーブルが作成される状態にする
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 5.1, 5.2, 5.3, 5.4, 6.2, 6.4, 6.5_

- [ ] 2. 配信の中核コンポーネントを実装する
- [ ] 2.1 正規メッセージと内容fingerprintを生成する
  - 件名と本文の空白入力を項目別に拒否し、本文の改行を保持する
  - `【件名】`、空行、本文の順で唯一の送信テキストを生成する
  - UTF-16コード単位で5,000以下を受理し、超過と孤立surrogateを検証エラーにする
  - formatter versionと入力・整形結果から曖昧性のない安定したSHA-256 fingerprintを生成する
  - 同じ有効入力は常に同じ整形結果とfingerprintになり、無効入力では送信可能な結果が生成されない状態にする
  - _Requirements: 1.2, 1.3, 1.4, 1.5_

- [ ] 2.2 確認済み内容だけを最終送信へ進める
  - 正規化済み内容のversionとfingerprintだけを署名したopaqueな確認トークンを発行する
  - トークン改変、version差異、件名または本文の変更を確認エラーとして拒否する
  - トークンへ本文、宛先、秘密値、操作IDを含めず、確認失敗時に送信試行や外部呼び出しを作らない
  - previewと同一内容だけが検証を通過し、編集後の内容は再確認が必要になる状態にする
  - _Requirements: 2.1, 2.2, 2.3, 2.4_

- [ ] 2.3 (P) LINEのpush結果を安全な内部結果へ変換する
  - 固定宛先へテキスト1件だけを送り、操作IDをLINE retry keyとしてSDK呼び出しを最大1回実行する
  - access tokenまたは固定宛先不足を外部呼び出し前に設定失敗へ変換する
  - 成功と409既受理からrequest IDを大文字小文字非依存で取得し、その他のHTTP statusを閉じた失敗種別へ分類する
  - connect 3秒・read 10秒の有限timeoutを設定し、SDKの自動retryを無効化してtimeoutを結果不明として返す
  - raw response本文、例外文字列、秘密値、宛先値を戻り値や通常ログへ出さず、安全な結果だけが上位層へ渡る状態にする
  - _Requirements: 3.1, 3.3, 3.5, 4.5, 5.3, 5.6, 6.1, 6.2, 6.3, 6.4, 6.5_
  - _Boundary: LINEGateway_
  - _Depends: 1.1_

- [ ] 2.4 (P) Frontendの配信DTOを安全に判別できるようにする
  - preview、send、status、共通エラーの閉じた通信結果を、成功・処理中・失敗・結果不明で判別できる形にする
  - 宛先、access token、channel secret、Backend内部型をFrontend契約へ含めない
  - 想定外の応答を無条件に信頼せずprotocol errorとして扱える境界を用意する
  - 各statusで必須情報が欠けた応答を利用可能な配信結果として扱わない状態にする
  - _Requirements: 3.2, 5.5, 5.6, 6.2, 6.4, 6.5_
  - _Boundary: DTO Types_

- [ ] 2.5 Frontendの合法な画面状態遷移を定義する
  - 入力、確認、送信中、処理中、通信結果不明、状態確認中、成功、失敗を排他的に表現する
  - 確認後の編集で整形テキスト、確認トークン、操作IDを破棄し、入力へ戻る操作では件名と本文を保持する
  - 最終送信開始時に操作IDを一度だけ生成し、送信中と状態確認中の重複操作を拒否する
  - network error後は元の操作IDとpayloadを保持し、404を確認した場合だけ同一操作の明示的な再試行を許可する
  - terminal結果から新しい配信を始めると前回の操作IDを破棄した空の入力状態になる
  - 状態遷移テストから、編集後の古い確認や二重submitが到達不能であることを確認できる状態にする
  - _Requirements: 2.4, 2.5, 3.4, 4.1, 4.2, 4.3, 4.5, 6.2, 7.1, 7.2, 7.3, 7.4_

- [ ] 3. Backendの送信フローを統合する
- [ ] 3.1 冪等な送信受付と外部結果の確定を統括する
  - 検証済み内容を処理中の送信試行として短いtransactionで確定してから、transaction外でLINEを1回だけ呼ぶ
  - 同じ操作IDと内容には既存状態を返し、同じIDの異なる内容と別IDの同一処理中内容を安全に拒否する
  - 成功、確定失敗、結果不明の各結果を日時とrequest IDを含むterminal状態へcompare-and-setで一度だけ確定する
  - terminal更新が競合しても先に確定した状態を上書きせず、失敗や不明から自動再送しない
  - 新規操作ごとの外部呼び出しが最大1回となり、受付済み試行に追跡可能な結果が保存される状態にする
  - _Requirements: 3.1, 3.3, 4.1, 4.2, 4.3, 4.4, 4.5, 5.1, 5.2, 5.3, 5.4, 6.2, 6.3_
  - _Depends: 1.2, 2.1, 2.3_

- [ ] 3.2 単一試行の状態確認と期限切れ収束を実装する
  - 操作IDに対応する既存試行だけを読み取り、存在しない場合は新しい試行や外部送信を作らない
  - 30秒の処理期限内は現在のprocessing状態と受付・期限日時を返す
  - 期限後はLINEを呼ばず、processing expiredの結果不明へcompare-and-setで一度だけ確定する
  - gateway完了と期限切れ確定が競合しても先に成立したterminal状態を後続処理が上書きしない
  - 繰り返し状態確認しても外部呼び出し回数が増えず、期限切れ試行が安全なunknownへ収束する状態にする
  - _Requirements: 4.3, 4.5, 5.4, 5.5, 5.6, 6.2, 6.3_
  - _Depends: 1.2, 3.1_

- [ ] 3.3 preview・最終送信・状態確認のHTTP契約を公開する
  - JSON型、必須項目、UUIDを検証し、previewでは正規テキストと確認トークンだけを返す
  - 最終送信は内容と確認トークンの一致を検証してから送信サービスを呼び、未確認・変更済み・無効入力ではDBとLINEを変更しない
  - 新規結果、既存terminal、処理中、操作ID再利用、同一内容処理中を設計どおりのHTTP statusと安全な応答へ変換する
  - 状態確認は既存試行の読取または期限切れ確定だけを行い、存在しないIDには試行を作らず安全な404を返す
  - ローカル開発専用としてauthenticationとpermissionを要求せず、認証情報なしのrequestで3つのendpointを利用可能にする
  - 固定文言の共通エラー形式を使い、宛先、秘密値、raw外部error、例外文字列を応答へ含めない
  - app-local URLをルートの`/api/deliveries/`配下へ合成し、無認証requestでも3つのPOST endpointが公開契約どおり応答する状態にする
  - _Requirements: 1.2, 1.5, 1.6, 2.1, 2.2, 2.3, 2.4, 3.2, 4.3, 4.4, 5.5, 5.6, 6.2, 6.4, 6.5_
  - _Depends: 2.2, 3.1, 3.2_

- [ ] 4. Frontendの配信体験を統合する
- [ ] 4.1 Backend契約と型付きで通信する
  - preview、send、単一試行statusだけを相対URLで呼び、JSON応答をDTO境界で検査する
  - 非2xx応答を安全な共通エラーへ変換し、想定外shapeをprotocol errorとして扱う
  - sendのnetwork error後は状態確認だけを行い、自動再送や新しい操作IDの生成をしない
  - statusの404後に利用者が明示した場合だけ、元と同じ操作ID、内容、確認トークンで同一送信操作を再試行する
  - UIが成功・処理中・確定失敗・結果不明を型付き結果として区別できる状態にする
  - _Requirements: 2.1, 2.2, 2.3, 3.5, 4.3, 4.5, 5.5, 5.6, 6.2_
  - _Depends: 2.4, 3.3_

- [ ] 4.2 入力・確認・送信結果を一つの安全な操作フローとして表示する
  - 件名と本文の入力、項目別エラー、Backendが返した実送信テキストの確認を別操作として提供する
  - 確認から戻った入力を保持し、編集後は古い確認で最終送信できないようにする
  - 宛先値や宛先入力を表示せず、最終送信中と状態確認中は操作を無効化して進行状態を示す
  - 成功は確認済み内容がLINEに受理された結果として、失敗と結果不明は送信成功が確定していない結果として安全な概要とともに表示する
  - 処理中・通信結果不明では送信とは区別した状態確認を提供し、404後だけ同一操作の明示的な再試行を表示する
  - terminal結果から新しい配信へ進むと、空入力と新しい操作IDのフローを開始できる状態にする
  - _Requirements: 1.1, 1.2, 1.4, 2.1, 2.2, 2.3, 2.4, 2.5, 3.2, 3.4, 3.5, 4.2, 4.3, 4.5, 5.5, 5.6, 6.2, 6.4, 7.1, 7.2, 7.3, 7.4_
